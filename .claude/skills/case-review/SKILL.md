指定された case.json について、source に記載された公開情報との整合性を検証してください。

## 目的

case.json の記載が公開情報に照らして正確かどうかを監査する。
記事を魅力的にすることではなく、**公開情報で確認できる範囲を逸脱していないかを検証すること**が目的。

## レビューペルソナ

以下の5つのペルソナの観点すべてから検証すること。

### 1. ファクトチェッカー
- 出典URLと case.json の事実整合性を厳密に検証
- 主体・関係性・実施段階が正確か
- 明記されていない関係を推定していないか

### 2. ビジネスレビュアー（読者理解）
- 一般的なビジネスユーザー（非技術者の管理職・政策担当など）が60秒で事例を理解できるか
- summary（課題）と value_proposition（価値）を並べて読んで「何が問題で、どうPETsで解決され、どんな成果が出たか」が読み取れるか
- summary に解決策、value_proposition に課題背景、など役割混在がないか
- 事例切り出しの妥当性（複数事例の混ぜ合わせ・企業紹介の事例化でないか）
- title は `{組織名}：{事例内容}` 形式か（コロンは `:` `：` どちらでも可）

### 3. PETs技術レビュアー
- プライバシー保護技術の専門家視点で技術記述を検証
- 匿名化/仮名化/合成データ/差分プライバシー/秘密計算/連合学習等の区別が正確か
- 技術解釈に飛躍がないか
- technology_category の分類が適切か
- privacy_enhancement_method / safety_assurance_method / utility_evaluation_method の手法・パラメータ・評価指標が出典と一致するか

### 4. 技術広報レビュアー
記載対象組織の広報担当者になったつもりで、対外発表として不適切な表現がないかを確認する。
- **言い過ぎチェック**: 出典が「実証」なのに「実現」、「検討」なのに「導入」、「可能にする」なのに「達成」になっていないか
- **マーケティング的誇張**: 「業界初」「圧倒的」「革新的」などの強い形容が出典にない場合は避ける
- **暗黙の比較優位主張**: 他社・業界標準との比較主張が根拠なく書かれていないか
- **一面的な価値主張**: value_proposition が「〜に価値がある」と断定し、他の価値側面が存在しないかのような書き方になっていないか。「〜は本事例で示された価値の一つである」のような表現を推奨
- **主体の扱い**: 組織の役割・業種・立場を誤って記述していないか

### 5. アカデミックレビュアー（引用厳密性）
学術カンファレンスの査読者になったつもりで、主張と出典の対応を点検する。
- **主張-出典の追跡可能性**: summary, value_proposition, 技術3項目, figureノードラベル それぞれについて、どの source 項目（できれば note に記載された箇所）が根拠となっているか辿れるか。根拠不明な主張はフラグする
- **一次情報 vs 二次情報**: 組織自身のプレスリリース・査読論文等の一次ソースと、ニュースサイト等の二次ソースを区別する。二次ソースしかない場合は一次ソースからの乖離がないかを確認
- **表現強度と根拠強度の整合**: 解釈・推論の場合は「〜と考えられる」「〜が期待される」、出典の直接的記述の場合は「〜とされている」「〜と報告されている」。どちらかに不整合がある場合はフラグ
- **引用の網羅性**: sources[].note が空・もしくは文献タイトルだけになっていないか（どの記述の根拠かが note から追えるべき）
- **数値の追跡性**: 百分率・規模・精度などの数値は必ず出典のどこかに記載があるものだけを使っているか

## 手順

### 1. case.json を読み込む

$ARGUMENTS がパスなら直接読む。事例IDなら `public/cases/{ID}/case.json` を読む。

### 2. source のURLをすべて確認する

- source にURLが複数ある場合は **必ずそのすべてを参照** する
- 1つのURLだけ読んで判断しない
- URLごとに、どの記述の根拠になっているかを意識して確認する
- URLがアクセス不能なら、その旨を明記する

### 3. 以下の6観点で検証する

#### 3-1. 主体
- どの企業・組織が何をしたかが正確か
- 支援企業、提供企業、導入企業、共同研究先、発表主体を取り違えていないか
- organization フィールドに必要な主体が含まれているか

#### 3-2. 関係性
- 企業同士・組織同士・データ同士・技術同士の関係が明記通りか
- 明記のない連携関係、因果関係、データ連携関係を勝手に補っていないか
- 「同じページに出ている」「同じテーマで紹介されている」だけで関係ありとしていないか

#### 3-3. 実施段階
- 構想/検討/実証/研究/共同研究/PoC/提供開始/導入済み/運用中のどれか
- 「検討している」を「実施している」と書いていないか
- 「共同研究を開始」を「実運用している」と書いていないか
- 「可能にする」を「実際に達成した」と書いていないか

#### 3-4. 技術内容
- 匿名化(anonymization)と仮名化(pseudonymization)を混同していないか
- 合成データ(synthetic_data)がプライバシー保護目的か、データ拡張目的かを区別しているか
- 元記事にない技術名を補っていないか
- technology_category が事例の中核技術を正しく反映しているか

#### 3-5. 価値・成果
- 効果、成果、便益が元記事で確認できるか
- 期待や狙いを、実績のように書いていないか
- value_proposition が出典の表現を超えていないか

#### 3-5b. 課題と価値の可読性
- summary（課題セクション）と value_proposition（価値セクション）だけを読んで、事例の概要が何となく分かるか
- summary は課題の背景・状況に焦点を当てているか（PETsの解決策に触れていないか）
- value_proposition はPETs適用による解決と成果に焦点を当てているか
- 両セクションの間に内容の重複や矛盾がないか
- ビジネスユーザーが「何が問題で、どう解決されたか」を自然に読み取れるか

#### 3-6. 事例の切り出し方
- 1件の case として切り出すことが妥当か
- 本来は別事例であるものを混ぜていないか
- 単なる企業紹介や技術紹介を無理に事例化していないか

#### 3-7. domain 分類
- domain フィールドが事例の主体企業・適用分野と一致しているか

#### 3-8. 発生時期 (occurred_at)
- occurred_at が出典記事の日付や記事中に記載された時期と整合しているか
- 出典記事の公開日・事例の実施時期と大きく乖離していないか
- 不明な場合は null のままで問題ないが、出典から明らかに特定できる場合は指摘する

### 4. figures（概要フロー）の検証

- nodes の各ラベルが出典で確認できる内容か
- 出典にない因果関係や成果を nodes に含めていないか
- category（source/constraint/process/application/outcome）が適切か

## 最重要ルール（厳守）

1. **明記されていない関係は推定しない** — 文脈的に関係がありそうでも、直接の記載がなければ認めない
2. **弱い表現は弱いまま扱う** — 「検討」「目指す」「可能にする」は「実施」「実現」「達成」ではない
3. **根拠がない場合は「根拠不足」として扱う** — 誤りと断定できないが確認できない記述
4. **推測で補完しない** — 不足部分を常識や業界知識で埋めない
5. **出典依存で評価する** — source に記載された公開情報のみで判断。他の一般知識で補強しない

## 出力形式

以下の形式でレビュー結果を出力する。

```markdown
## レビュー結果: {事例ID}

### 総合判定
正確 / 概ね正確 / 要修正 / 根拠不足が多い / 事例として不適切

### 総評
（2〜6文）

### source確認結果
| URL | 参照可否 | 主な用途 | 備考 |
|---|---|---|---|

### 項目別レビュー
| 項目 | case.jsonの記載 | 判定 | 根拠URL | 指摘内容 | 修正案 |
|---|---|---|---|---|---|

判定: 正確 / 概ね正確 / 要修正 / 根拠不足 / 判定不能

### ペルソナ別判定
| ペルソナ | 判定 | 主な指摘 |
|---|---|---|
| 1. ファクトチェッカー | OK / 懸念あり / 要修正 | |
| 2. ビジネスレビュアー | OK / 懸念あり / 要修正 | |
| 3. PETs技術レビュアー | OK / 懸念あり / 要修正 | |
| 4. 技術広報レビュアー | OK / 懸念あり / 要修正 | |
| 5. アカデミックレビュアー | OK / 懸念あり / 要修正 | |

### 特に重要な懸念
- 関係性の過剰推定: あり/なし（内容）
- 実施段階の言い過ぎ: あり/なし（内容）
- 技術解釈の飛躍: あり/なし（内容）
- 価値記述の言い過ぎ・一面的断定: あり/なし（内容）
- マーケティング的誇張表現: あり/なし（内容）
- 主張-出典の対応が追跡不能: あり/なし（内容）
- 課題と価値だけで概要が伝わるか: はい/いいえ（内容）
- title が `{組織名}：{事例内容}` 形式か: はい/いいえ
- domain分類の誤り: あり/なし（内容）
- technology_category の誤り: あり/なし（内容）
- occurred_at の不整合: あり/なし（内容）
```

### 5. 修正が必要な場合

「要修正」以上の判定の場合は、公開情報で確認できる内容だけに限定した修正案を提示する。
断定を避けるべき箇所は、適切に弱めた表現にする。

## 今回のレビューで頻出した典型的な問題パターン

以下は過去のレビューで繰り返し検出された問題。特に注意して確認すること。

1. **anonymization / pseudonymization の混同**: 仮名加工情報を扱う事例が anonymization に分類されがち。法制度上の区別を正確に反映する
2. **domain 分類の誤り**: 主体企業の業種ではなく通信等の汎用カテゴリに誤分類されがち
3. **ベンダ一覧の技術情報を個別事例に過剰適用**: ベンダの一般的な技術能力を、特定事例で使用された技術として記載
4. **実施段階の過剰表現**: 「検討」「発表」「実証」を「導入」「運用中」「実現」に格上げ
5. **出典が方針公表レベル**: プライバシーポリシー等からの事例化で、実際の活用実績が未確認
6. **organization の欠落**: 連名発表の一部主体が organization に含まれていない
7. **将来展望と実績の混同**: 出典の期待・展望を成果・実績として記述
